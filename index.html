Hermes Agent × Local AI

A complete, copy-pasteable guide to running Hermes Agent with a local MiniMax M3 model on your engineering firm's PC — with SSH access, GitHub, office tools, and automated backups.

● Live Open-Source No API Keys Self-Hosted 1M Context

↗ Architecture Overview

Everything runs on a single powerful PC. You access it from your laptop, phone, or another machine via SSH, Telegram, or a web dashboard.

🧠

MiniMax M3 (Local LLM)

428B parameter MoE model (23B active). Served via vLLM or Ollama on the PC. Provides reasoning, tool-calling, and code generation without any cloud API.

⚡

Hermes Agent

The self-improving agent that connects to the local model. Handles file editing, terminal commands, web browsing, delegation, and cross-session memory.

🔗

SSH + Gateway

SSH from any device on the network. Optional Telegram/Discord gateway for remote input. Hermes CLI works identically over SSH.

📁

File Access & Backups

Syncthing for live sync, Restic for encrypted versioned backups, and a shared network drive for office suite document access.

🐙

GitHub Integration

Hermes connects to GitHub via SSH deploy keys and the GitHub CLI. Can open PRs, run CI checks, create repos, and manage issues.

📋

Office Suite

OnlyOffice or LibreOffice for document editing. Shared directories across the team. Hermes generates .docx, .xlsx, .pptx files on demand.

1 Set Up the Local AI phase 1/6

Install Ollama, pull MiniMax M3, and verify the model works — then configure it to serve an OpenAI-compatible API.

1.1 Install Ollama 1.2 Pull M3 1.3 64K Context 1.4 Verify API

1.1 — Install Ollama

Bash
curl -fsSL https://ollama.com/install.sh | sh
# Verify
ollama --version
curl http://localhost:11434/api/tags

1.2 — Pull MiniMax M3

Bash
# Pull the M3 model (Quantized Q4_K_M — best quality/speed balance)
ollama pull minimax-m3:q4_K_M

# Verify it loaded correctly
ollama ps
⚡ Alternative: vLLMFor production serving with higher throughput, use vLLM instead: pip install vllm && python -m vllm.entrypoints.openai.api_server --model minimax/minimax-m3 --tensor-parallel-size 2 --port 8000

1.3 — Create 64K Context Model

Default Ollama context is 2048 tokens — way too small. Hermes needs 64K+ for agentic tasks.

Bash
cat > ~/Modelfile-m3-64k << 'EOF'
FROM minimax-m3:q4_K_M
PARAMETER num_ctx 64000
PARAMETER num_gpu 99
EOF

ollama create m3-64k -f ~/Modelfile-m3-64k

1.4 — Verify the API Works

Bash
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "m3-64k",
    "messages": [{"role": "user", "content": "Say hello and identify yourself."}],
    "max_tokens": 100
  }'
✅ DoneIf you see a JSON response with "MiniMax" in the content, your local AI is running.

2 Install & Configure Hermes phase 2/6

Install Hermes Agent, configure it to use your local M3 model, and set up skill management.

Bash
# Install Hermes via the recommended method
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | sh

# Or via pip
# pip install hermes-agent

# Verify
hermes --version
🔧 ConfigurationEdit ~/.hermes/config.yaml to point to your local M3 model. Two ways to do this:

Option A: Interactive Setup

Bash
hermes setup
# → Select "Custom Endpoint"
# → Enter: http://localhost:11434/v1
# → API key: no-key (or any placeholder)
# → Model: m3-64k

Option B: Manual Config

Write this to ~/.hermes/config.yaml:

YAML
model:
  default: "m3-64k"
  provider: "custom"
  base_url: "http://localhost:11434/v1"

# Optional: Fallback to a faster model for simple queries
  fallback:
    provider: "custom"
    base_url: "http://localhost:11434/v1"
    model: "llama3.2:3b"

Keep the Model Loaded

Ollama unloads models after 5 minutes of inactivity. Keep M3 hot:

Bash
sudo mkdir -p /etc/systemd/system/ollama.service.d
sudo tee /etc/systemd/system/ollama.service.d/override.conf << 'EOF'
[Service]
Environment="OLLAMA_KEEP_ALIVE=24h"
EOF
sudo systemctl daemon-reload
sudo systemctl restart ollama
✅ DoneRun hermes to start a session. It should respond using your local M3 with zero cloud calls.

3 SSH + Network Access phase 3/6

Make the PC accessible from your laptop, phone, or a colleague's machine — securely.

Bash
# Install & enable SSH server
sudo apt install openssh-server
sudo systemctl enable --now ssh

# Set up firewall — only allow SSH from your LAN
sudo ufw allow from 192.168.1.0/24 to any port 22 proto tcp
# (Adjust subnet to match your network)
🔒 SecurityNever expose SSH directly to the internet. Use a VPN (Tailscale, WireGuard) or a bastion host for remote access. Add your SSH key instead of using passwords:
Bash (from your laptop)
ssh-copy-id user@hermes-pc-ip
# Then disable password auth on the PC:
sudo sed -i 's/^#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
# Or: sudo sed -i 's/^PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart ssh

Using Hermes over SSH

Bash
# From your laptop — Hermes runs on the PC, you control it remotely
ssh user@hermes-pc
hermes
# Now you're talking to Hermes via the local M3. All files are on the PC.
✅ DoneTest with: ssh user@hermes-pc "ollama ps && hermes --version"

4 GitHub Integration phase 4/6

Connect Hermes to your org's GitHub. Deploy keys, CLI auth, and CI/CD pipeline setup.

Bash
# Install GitHub CLI
curl -fsSL https://cli.github.com/packages/githubcli-archive-keyring.gpg | \
  sudo dd of=/usr/share/keyrings/githubcli-archive-keyring.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/githubcli-archive-keyring.gpg] https://cli.github.com/packages stable main" | \
  sudo tee /etc/apt/sources.list.d/github-cli.list > /dev/null
sudo apt update && sudo apt install gh -y

# Authenticate (device flow)
gh auth login --hostname github.com --git-protocol ssh

# Verify
gh auth status
gh repo list

Deploy Key for CI/CD (Read-Only)

Bash
# Generate a deploy key for Hermes
ssh-keygen -t ed25519 -C "hermes-agent@firm" -f ~/.ssh/hermes-deploy -N ""
cat ~/.ssh/hermes-deploy.pub
# → Add this as a deploy key in GitHub → repo → Settings → Deploy Keys

# Configure SSH to use it
cat >> ~/.ssh/config << 'EOF'
Host github.com
  HostName github.com
  User git
  IdentityFile ~/.ssh/hermes-deploy
EOF

Clone Your Repos

Bash
# Clone all org repos (example)
mkdir -p ~/code
cd ~/code
gh repo list your-org --limit 50 --json nameWithOwner -q '.[].nameWithOwner' | \
  while read repo; do gh repo clone "$repo"; done

5 Office Suite & File Access phase 5/6

Document editing, shared drives, and easy file access for the engineering team.

📄

OnlyOffice

Self-hosted document server with real-time collaboration. Compatible with .docx, .xlsx, .pptx. Accessible via web browser.

🔄

Syncthing

Live peer-to-peer file sync between the PC and team laptops. No cloud middleman. Encrypted in transit.

🗄️

Samba / NFS

Traditional network share. Mounts as a drive on Windows/Mac/Linux. Accessible from LibreOffice, Excel, Word.

Option: Samba Share (Simplest)

Bash
sudo apt install samba -y
sudo mkdir -p /srv/shared/{documents,spreadsheets,presentations,projects}
sudo chmod -R 775 /srv/shared
sudo chown -R :users /srv/shared

sudo tee -a /etc/samba/smb.conf << 'EOF'

[Shared]
   path = /srv/shared
   browseable = yes
   read only = no
   guest ok = no
   valid users = @users
   force group = users
   create mask = 0664
   directory mask = 0775
EOF

# Add users
sudo smbpasswd -a your-username
sudo systemctl restart smbd

6 Backup Protocols phase 6/6

Automated, encrypted, versioned backups. Restic to local + remote storage. The 3-2-1 rule.

📐 The 3-2-1 Rule3 copies of your data, on 2 different media, with 1 copy off-site. This setup achieves all three.
Bash
# Install Restic
sudo apt install restic -y

# Initialize backup repository (local + offsite)
restic init --repo /mnt/backup/restic-repo
# Set a strong encryption password and save it in a password manager!

# If you have a remote backup target (S3, Backblaze, rsync.net):
# restic init --repo s3:https://s3.eu-central-1.amazonaws.com/your-bucket/hermes-backup

Backup Script

Save as ~/backup.sh — covers code, config, and documents:

Bash
#!/bin/bash
# ~/backup.sh — Daily backup for Hermes engineering PC
set -e

REPO="/mnt/backup/restic-repo"
PASSWORD_FILE="$HOME/.restic-password"
BACKUP_PATHS=(
  "$HOME/code"           # All repositories
  "$HOME/.hermes"        # Hermes config, memory, skills
  "/srv/shared"          # Shared office documents
  "$HOME/.ssh"           # SSH keys (encrypted)
  "/etc/ollama"          # Ollama config
  "/etc/nginx"           # Web server config if used
)

# Prune policy: keep 7 daily, 4 weekly, 6 monthly
restic -r "$REPO" \
  --password-file "$PASSWORD_FILE" \
  backup "${BACKUP_PATHS[@]}" \
  --exclude="**/node_modules" \
  --exclude="**/.venv" \
  --exclude="**/__pycache__" \
  --tag "daily-$(date +%Y-%m-%d)"

restic -r "$REPO" \
  --password-file "$PASSWORD_FILE" \
  forget --keep-daily 7 --keep-weekly 4 --keep-monthly 6 \
  --prune

# Optional: copy to off-site
# restic copy --from-repo "$REPO" --to-repo s3:https://s3....

echo "✅ Backup complete: $(date)"
Bash
chmod +x ~/backup.sh

# Set up daily cron
(crontab -l 2>/dev/null; echo "0 2 * * * /home/user/backup.sh >> /var/log/hermes-backup.log 2>&1") | crontab -

# Or use Hermes' own cron system:
# > "Schedule: daily backup at 2 AM"

What's Excluded (and Why)

PathWhyHow to Handle
~/.ollama/models/Model weights are huge (20-200 GB), can be re-downloadedSkip — just backup /etc/ollama config
node_modules/Easily reinstalled, bloated backupsExcluded via --exclude
.venv/ / __pycache__Python cache, recreatableExcluded via --exclude

⚠ Troubleshooting

Common issues and their fixes — from a Firmware Engineer's perspective.

🔥 Ollama won't start / GPU not detected
Check:
nvidia-smi — should show your GPU and driver version.
sudo systemctl status ollama — check for CUDA errors.

Fix:
- Install NVIDIA container toolkit: sudo apt install nvidia-container-toolkit
- Ensure OLLAMA_GPU_OVERHEAD=1 is set
- Check journalctl -u ollama -n 50 --no-pager for detailled errors
🐢 Model is extremely slow (under 2 t/s)
Likely cause: CPU-only inference or GPU offloading isn't working.

Check: ollama ps — see the "layers" column. You want most layers on GPU.
Fix:
- Set num_gpu 99 in your Modelfile and recreate the model
- If you have multiple GPUs, enable tensor parallelism: OLLAMA_NUM_GPU=2
- Try a smaller quantization: ollama pull minimax-m3:q3_K_M
- Widen API timeout: add HERMES_API_TIMEOUT=1800 to ~/.hermes/.env
🔗 Hermes can't connect to the model
Check:
- curl http://localhost:11434/api/tags — is Ollama reachable?
- curl http://localhost:11434/v1/chat/completions ... — is the API compatible?
- Review ~/.hermes/config.yaml — correct base_url and model name?

Fix:
- The model name in config must match ollama list output exactly
- If using the custom provider, set provider: "custom" and base_url: "http://localhost:11434/v1"
- Try hermes setup again and choose "Custom Endpoint"
💾 Out of memory / OOM kills
Symptoms: Ollama process killed, "killed" in journalctl, system freeze.

Fix:
- Try a smaller quantization: ollama pull minimax-m3:q3_K_M or q2_K
- Reduce context size: change num_ctx 64000 to num_ctx 32000
- Add swap space: sudo fallocate -l 64G /swapfile && sudo chmod 600 /swapfile && sudo mkswap /swapfile && sudo swapon /swapfile
- Monitor with htop and nvidia-smi -l 2
🌐 Can't SSH from outside the LAN
Don't: Port-forward SSH directly to the internet.

Do:
- Tailscale (recommended): curl -fsSL https://tailscale.com/install.sh | sh && sudo tailscale up
- WireGuard: sudo apt install wireguard && wg-quick up ./wg0.conf
- Cloudflare Tunnel: sudo cloudflared tunnel create + route to SSH via dashboard

Then SSH to user@tailscale-ip or via the tunnel.
🐙 GitHub CLI auth fails
Fix:
- Run gh auth login with the --web flag for device flow
- Ensure SSH agent is running: eval "$(ssh-agent -s)" && ssh-add ~/.ssh/hermes-deploy
- Test: ssh -T git@github.com — should say "Hi your-org/repo"
- If using HTTPS instead: gh auth setup-git
🔄 Restore from a backup
# List snapshots
restic -r /mnt/backup/restic-repo snapshots

# Restore latest to a directory
restic -r /mnt/backup/restic-repo restore latest --target ~/restore/

# Or mount and browse like a filesystem
restic -r /mnt/backup/restic-repo mount /mnt/restore-mount
📡 Hermes Telegram/Discord gateway won't connect
Fix:
- Enable the platform in ~/.hermes/config.yaml:
platforms:
  telegram:
    enabled: true
    token: "YOUR_BOT_TOKEN"
  discord:
    enabled: true
    token: "YOUR_BOT_TOKEN"
- Run hermes gateway
- Check hermes gateway --status for connection errors
- Ensure the PC has outbound HTTPS access to Telegram/Discord APIs

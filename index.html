↗ Architecture Overview

Everything runs on a single powerful PC — either a GPU workstation or a Beelink mini PC with 128 GB unified memory. You access it from any device via SSH, Telegram, Discord, or a web dashboard. Models run locally, via cloud API, or both in a hybrid setup.

🧬

GLM 5.2 (Z.ai / Local)

#1 open-source reasoning model. 744B MoE (40B active), 1M token context, MIT license. Runs via Z.ai cloud API on any hardware, or self-host on enterprise GPUs. Coding, reasoning, long-horizon agent tasks.

🧠

MiniMax M3 (Local LLM)

428B parameter MoE (23B active). Served via Ollama or vLLM. Fits in 128 GB RAM at Q2_K (~107 GB). Reasoning, tool-calling, code generation — no cloud API needed.

⚡

Hermes Agent

The self-improving agent that connects to any model. Handles file editing, terminal commands, web browsing, delegation, and cross-session memory. Provider-agnostic — switch between models freely.

🔗

SSH + Gateway

SSH from any device on the network. Optional Telegram/Discord gateway for remote input. Hermes CLI works identically over SSH — whether you're on-site or at NILT.

📁

File Access & Backups

Syncthing for live sync, Restic for encrypted versioned backups, Samba/NFS network share. Access your code and documents from any machine.

🐙

GitHub Integration

Connects via SSH deploy keys + GitHub CLI. Open PRs, run CI checks, create repos, manage issues — all through Hermes.

1.1 — Install Ollama

Bash
curl -fsSL https://ollama.com/install.sh | sh
# Verify
ollama --version
curl http://localhost:11434/api/tags

1.2 — Pull Your Models

Choose what fits your hardware. You can pull all of them — Ollama manages VRAM/RAM automatically.

Bash
# Option A: MiniMax M3 (428B, needs ~107 GB at Q2_K)
# Best for: Full agentic Hermes, code generation, complex reasoning
ollama pull minimax-m3:q2_K

# Option B: GLM-5 (744B, needs 2-bit quant ~241 GB — requires enterprise GPU)
# If you have 8× H100 or similar: ollama pull glm-5:q2_K

# Option C: Gemma 4 models (Apache 2.0, fits any hardware)
ollama pull gemma4:26B-A4B    # MoE 26B, 14 GB — excellent all-rounder
ollama pull gemma4:31B        # 31B, 17.5 GB — heavy reasoning
ollama pull gemma4:E4B        # 4.5B, fast companion model

# Option D: Lightweight fallback
ollama pull llama3.2:3b       # 3B, 2 GB — always hot, ultra-fast

1.3 — Create 64K Context Variants

Default Ollama context is 2048 tokens. Hermes needs 64K+ for agentic tasks. Create a custom variant for each model you pulled.

Bash
# For MiniMax M3
cat > ~/Modelfile-m3-64k << 'EOF'
FROM minimax-m3:q2_K
PARAMETER num_ctx 64000
PARAMETER num_gpu 99
EOF
ollama create m3-64k -f ~/Modelfile-m3-64k

# For Gemma 4 26B
cat > ~/Modelfile-g4-64k << 'EOF'
FROM gemma4:26B-A4B
PARAMETER num_ctx 64000
PARAMETER num_gpu 99
EOF
ollama create g4-64k -f ~/Modelfile-g4-64k

# Verify both exist
ollama list

1.4 — Connect GLM 5.2 via Cloud API

GLM 5.2 (744B) is too large even for 128 GB RAM at full precision. But it's available as a cloud API from Z.ai with the full 1M context — and it's MIT open-source. This works on any machine, including the Beelink.

Bash
# Create a Hermes profile for GLM 5.2
hermes profile create glm

# Configure it to use Z.ai's API
hermes -p glm config set model.default "glm-5.2"
hermes -p glm config set model.provider "zai"
hermes -p glm config set model.base_url "https://api.z.ai/api/coding/paas/v4"
hermes -p glm config set model.context_length 1000000

# Set your API key (get one at https://z.ai)
export Z_AI_API_KEY="z_ai_your_key_here"

# Smoke test
hermes -p glm -z "Say ready and nothing else."
🧬 GLM 5.2 Specs744B total / 40B active parameters — MoE architecture. MIT license (fully open-source). 1M token context window. Best-in-class on SWE-Bench and coding benchmarks. ~$0.95/M input tokens via API. When enterprise hardware is available, self-host via vLLM or SGLang.

1.5 — Verify All APIs Work

Bash
# Test local Ollama models
curl http://localhost:11434/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "m3-64k", "messages": [{"role": "user", "content": "Identify yourself."}], "max_tokens": 100}'

# Test GLM 5.2 cloud API
hermes -p glm -z "What model are you running?"

# List all available

2 Install & Configure Hermes phase 2/6

Install Hermes Agent, configure it for multi-model routing, and set up automatic fallback between your local models and GLM 5.2.

Bash
# Install Hermes
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | sh
hermes --version
🔧 Multi-Model ConfigurationThis config routes tasks to the best model automatically. Local M3 or Gemma for everyday work, GLM 5.2 for complex reasoning and long context tasks.

Option A: Interactive Setup

Bash
# Run the setup wizard
hermes setup
# → Select "Custom Endpoint"
# → Enter: http://localhost:11434/v1
# → Model: m3-64k (or g4-64k for Gemma 4)

Option B: Manual Multi-Model Config

Write this to ~/.hermes/config.yaml:

YAML
model:
  # Primary: local model for everyday agentic work
  default: "m3-64k"
  provider: "custom"
  base_url: "http://localhost:11434/v1"
  api_key: "no-key"

  # Fallback chain — Hermes tries each on failure or overload
  fallback:
    - provider: "custom"
      base_url: "http://localhost:11434/v1"
      model: "g4-64k"
      api_key: "no-key"
    - provider: "custom"
      base_url: "http://localhost:11434/v1"
      model: "gemma4:E4B"
      api_key: "no-key"

# GLM 5.2 profile (separate, switch with hermes profile use glm)
# Create via: hermes profile create glm
# Then:     hermes -p glm config set ...

# Keep local models hot
auxiliary:
  compression:
    model: "gemma4:E4B"
    provider: "custom"
    base_url: "http://localhost:11434/v1"
    api_key: "no-key"

Keep Models Loaded (24h keep-alive)

Bash
sudo mkdir -p /etc/systemd/system/ollama.service.d
sudo tee /etc/systemd/system/ollama.service.d/override.conf << 'EOF'
[Service]
Environment="OLLAMA_KEEP_ALIVE=24h"
EOF
sudo systemctl daemon-reload
sudo systemctl restart ollama
✅ DoneRun hermes — should respond using your local model. Switch to hermes -p glm for GLM 5.2 with 1M context.

Bash

[Shared]
   path = /srv/shared
   browseable = yes
   read only = no
   guest ok = no
   valid users = @users
   force group = users
   create mask = 0664
   directory mask = 0775
EOF

sudo smbpasswd -a your-username
sudo systemctl restart smbd

6 Backup Protocols phase 6/6

Automated, encrypted, versioned backups. Restic to local + remote storage. The 3-2-1 rule.

📐 Beelink NoteWith the dual 10GbE ports, backup to a NAS or second NVMe at 1 GB/s. The 2 TB NVMe is fast enough that backup doesn't slow down model inference.
Bash
sudo apt install restic -y
restic init --repo /mnt/backup/restic-repo

# Create backup script at ~/backup.sh
cat > ~/backup.sh << 'SCRIPT'
#!/bin/bash
REPO="/mnt/backup/restic-repo"
restic -r "$REPO" backup \
  "$HOME/code" \
  "$HOME/.hermes" \
  "/srv/shared" \
  "$HOME/.ssh" \
  "/etc/ollama" \
  --exclude="**/node_modules" \
  --exclude="**/.venv" \
  --tag "daily-$(date +%Y-%m-%d)"

restic -r "$REPO" forget \
  --keep-daily 7 --keep-weekly 4 --keep-monthly 6 --prune
SCRIPT

chmod +x ~/backup.sh
(crontab -l 2>/dev/null; echo "0 2 * * * /home/user/backup.sh >> /var/log/hermes-backup.log 2>&1") | crontab -

🏢 Access from Another PC (NILT Edition) production

How colleagues access Hermes from their workstations — at NILT, from home, or anywhere with internet. No port forwarding, no security holes.

🔒 Never Expose DirectlyHermes and Ollama should never be exposed to the public internet directly. Use one of these secure methods instead.

Method A: Tailscale (Recommended — Zero Config)

Creates a private mesh network. Both PCs get a virtual IP. Works through firewalls, NAT, and CGNAT. No open ports needed.

Bash (on Hermes PC)
curl -fsSL https://tailscale.com/install.sh | sh
sudo tailscale up
# → Visit the URL to auth. Note the Tailscale IP (e.g., 100.x.x.x)
sudo tailscale ip -4
Bash (on your NILT laptop)
# From NILT — SSH via Tailscale
ssh user@100.x.x.x

# Now run Hermes
hermes

# Or run a single command
ssh user@100.x.x.x "hermes -z 'Read the README and summarize it'"

Method B: Cloudflare Tunnel (No VPN Needed)

Expose SSH via Cloudflare Tunnel — users connect through Cloudflare's edge network. No public IP needed, DDoS protection built in.

Bash (on Hermes PC)
# Install cloudflared
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64 -o /usr/local/bin/cloudflared
chmod +x /usr/local/bin/cloudflared

# Authenticate and create a tunnel
cloudflared tunnel login
cloudflared tunnel create hermes-ssh

# Configure the tunnel (edit ~/.cloudflared/config.yml)
# url: ssh://localhost:22
# Run it:
cloudflared tunnel run hermes-ssh

Method C: Telegram / Discord Gateway (Zero Client Setup)

No SSH needed at all. Hermes connects to Telegram/Discord via the gateway — you message it like a chatbot from your phone or any computer.

Bash (on Hermes PC)
# Enable Telegram in ~/.hermes/config.yaml
# platforms:
#   telegram:
#     enabled: true
#     token: "YOUR_BOT_TOKEN"

# Start the gateway
hermes gateway

# Now message your bot on Telegram — Hermes responds instantly
🏢 NILT Workflow Example
1. Hermes PC runs at the office with GLM 5.2 + local models
2. You SSH in from NILT via Tailscale: ssh hermes@100.x.x.x
3. Ask Hermes: "Run the CI pipeline on project-x and report back"
4. Hermes responds with results — all commands run on the PC, you see the output
5. Or just message the Telegram bot from the lab floor — no terminal needed

Security Checklist for NILT Access

  • Use Tailscale or Cloudflare Tunnel — never expose SSH directly
  • Disable password auth: PasswordAuthentication no
  • Use SSH keys only — one per team member
  • Set up ~/.ssh/authorized_keys with only approved keys
  • Consider MFA via Tailscale ACLs for production
  • Monitor access: journalctl -u ssh -n 20 to check logins
  • Backup ~/.hermes daily — it holds memory, skills, and config

⚠ Troubleshooting

Common issues and their fixes — from a Firmware Engineer's perspective.

🔥 Ollama won't start / GPU not detected (Beelink / AMD)
Check:
sudo journalctl -u ollama -n 50 --no-pager — look for GPU errors
sudo apt install rocm-hip-libraries clinfo — install ROCm/Vulkan support

Fix:
- Set VRAM split to 96 GB in BIOS first
- sudo apt install mesa-vulkan-drivers
- Set OLLAMA_GPU_OVERHEAD=1 in ~/.hermes/.env
- Try: OLLAMA_VULKAN_VISIBLE_DEVICES=1 ollama serve
